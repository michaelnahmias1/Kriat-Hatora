## עדכון v6.1 — שלב 5 מומש: מסלול ה-AI עבר מ-Colab לתוך האפליקציה (17.7)

המטרה של שלב 5 ("להעלים את Colab מהזרימה היומיומית") מומשה עם **Modal** —
worker עם GPU T4 לפי-שימוש (הקרדיט החודשי של התוכנית החינמית מכסה שימוש אישי;
הצריכה הצפויה ~1-2$ לחודש).

1. **עקרון מנחה (החלטת בעל הפרויקט): שום שינוי בשיטה שעובדת.** משיכת הטקסט
   מ-Sefaria נשארת דרך אותו API בדיוק, ועיבוד ההקלטה וחלוקת הזמנים נשארים
   באותו צינור היברידי בדיוק — `modal_worker/app.py` הוא עטיפה בלבד שקוראת
   ל-`run_hybrid` הקיים ללא נגיעה ב-`src/aligner`.
2. **פרוטוקול מול הטלפון:** ‏`POST /submit` (הקלטה + פרמטרים + token) מחזיר
   מיד call_id, והדפדפן שואל `GET /status` כל ~5 שניות — לא מחזיקים בקשת HTTP
   פתוחה דקות על חיבור סלולרי. ההעלאה הולכת ישירות ל-Modal (מגבלת ~4.5MB לגוף
   בקשה ב-Vercel ממילא לא מאפשרת לתווך אותה).
3. **נגד cold-start:** המודלים (whisper של ivrit.ai + ‏MMS_FA) נאפים בתוך
   ה-image בזמן הבנייה; בסיס ה-image הוא CUDA+cuDNN מערכתי כמו ב-Colab.
4. **באפליקציה:** כפתור "🛰️ תזמון AI מדויק בענן" בשלב 3; הזמנים ודוח האיכות
   נכנסים ישר לשלב הבדיקה (אותו מנגנון של ייבוא ה-SRT הידני, שחולץ לפונקציה
   משותפת). כשקבועי ה-endpoint ריקים הכפתור מוסתר — האפליקציה עובדת כמו קודם.
5. **Colab נשאר כגיבוי מלא** (worker.ipynb לא נגע) — אותו צינור, שתי סביבות הרצה.
6. **אבטחה בסיסית:** ‏token משותף (secret ב-Modal) כהגנת-עלות; מתועד שהוא חשוף
   בצד הלקוח ומספיק לכלי אישי בלבד.

## עדכון v5.2 — הממשק הקבוע נבנה ונפרס (16.7)

1. **הממשק הקבוע חי: ‏https://kriat-hatora.vercel.app** — פרויקט Vercel חדש ונפרד בשם
   `kriat-hatora` (הפרויקטים האחרים בחשבון לא נגעו). דף אחד בעברית, RTL, מותאם לטלפון:
   בחירת ספר/פרק/טווח ← טקסט MAM חתוך לפי טעמים ← בחירת הקלטה מקומית ← תזמון ← SRT.
2. **הפתרון לבעיית Supabase: לא צריך Supabase.** התוכנית החינמית מוגבלת ל-2 פרויקטים
   ושניהם תפוסים — אז הארכיטקטורה שנבנתה מוותרת על אחסון ענן לגמרי: ההקלטה נשארת
   בטלפון (נטענת ישירות לדפדפן), התזמון וה-SRT נוצרים בצד הלקוח, ורק משיכת הטקסט
   מ-Sefaria רצה בפונקציית שרת קלה (Python, ספריה סטנדרטית) על Vercel. אם בעתיד
   יידרש אחסון (ריבוי משתמשים, היסטוריה) — ‏Vercel Blob/KV באותו חשבון, בלי Supabase.
3. **תזמון בלי GPU: מצב הקשות (tap-along).** מאזינים להקלטה בתוך הדף ומקישים ברגע
   שכל מקטע מתחיל (עם פיצוי זמן-תגובה מוגדר, ברירת מחדל 300ms, ומסך שנשאר ער).
   זה נותן סנכרון של עשיריות שנייה בלי AI — ועונה על דרישת "הכל מהטלפון" במלואה.
   בנוסף קיים מצב חלוקה אוטומטית לפי אורך הטקסט (משוקלל אותיות + הפסקות סוף-פסוק).
   מסלול היישור המדויק ב-Colab (worker.ipynb) נשאר כמסלול ה-AI.
4. **תוקן באג אמיתי שהתגלה בבדיקת האמת מול MAM:** ‏Sefaria עוטף אותיות מיוחדות בתגית
   HTML בתוך המילה (האות הגדולה של בְּרֵאשִׁית). ההחלפה הישנה של תגיות ברווח שברה את
   המילה לשתיים. עכשיו תגיות עיצוב מוסרות בלי רווח, ‏`<br>` הופך לרווח, והערות שוליים
   (`<sup>`/`<i class="footnote">`) מוסרות עם תוכנן. התיקון ב-`src/chunker` משרת גם
   את מסלול ה-Colab.

## עדכון v6 — הפייפליין ההיברידי (16.7)

יישור CTC יחיד על הקלטה שלמה התברר כנקודת התורפה שנצפתה בסעיף הסיכונים: מליסמה
אחת שהמודל לא מבין מסיטה את כל ההמשך, ואין התאוששות. v6 מחליף את מסלול היישור
(שלב 5 בצינור) בשלוש שכבות שכל אחת מפצה על חולשת קודמתה — **בלי לגעת בטקסט
המוצג, שנשאר תמיד ה-display המדויק מ-Sefaria (מנוקד ומוטעם)**:

1. **תמלול (עוגנים):** faster-whisper עם מודל עברי של ivrit.ai נותן חותמות-זמן
   גסות למילים. `anchors.py` מצליב אותן עם הטקסט הידוע (Needleman-Wunsch מונוטוני;
   מרחק-עריכה שמוזיל ו/י — כתיב חסר מול מלא; סינון הזיות בשער-הקשר ובקצב-קריאה)
   ובוחר "עוגנים" אמינים. התמלול לא קובע תזמון סופי ולא טקסט — עיגון בלבד.
2. **יישור כפוי ממושכן:** `windows.py` תוחם חלונות בין עוגנים עוקבים, ו-MMS_FA
   (המסלול הקיים, עכשיו `MmsAligner` רב-שימושי) רץ על כל חלון בנפרד. סחיפה חסומה
   לחלון; מגבלת 6 הדקות נעלמת. חלון שנכשל → מיזוג עם שכן דרך העוגן החלש וניסיון
   נוסף → ואם גם זה נכשל, אינטרפולציה משוקללת-אותיות + דגל.
3. **VAD ‏(silero):** `vad.py` מצמיד גבולות מקטעים לתחילות-דיבור סמוכות — סוף
   פסוק כמעט תמיד נושא נשימה אמיתית, ולכן גבול-פסוק מקבל רדיוס מוגדל וקדימות.

לכל מקטע מחושב **ציון ביטחון** (`confidence.py`) מארבעה אותות בלתי-תלויים, ולצד
ה-SRT נכתב דוח בעברית שמפרט אילו מקטעים לגרור ידנית ב-CapCut. רתמת ההערכה
(`evaluate.py` + `notebooks/poc_eval.ipynb`) מודדת את שלוש השיטות — baseline /
MMS טהור / היברידי — מול ground truth מסומן-ידנית, לפי מדדי שלב 2.

עקרון ההדרדרות החיננית: אם התמלול נכשל לגמרי על קריאה בטעמים (הסיכון המרכזי
בסעיף 6), מעט עוגנים = חלונות גדולים ≈ ההתנהגות הישנה — לעולם לא גרוע ממנה.

## 1. עקרונות מוצר (v5)

1. **משתמש אחד. הטקסט הוא הערך. המקטע הוא היחידה.** (ירושה מ-v3/v4, לא השתנה.)
2. **חדש: הטלפון הוא המכשיר היחיד שחובה.** כל שלב בזרימה חייב להיות אפשרי מדפדפן/אפליקציות בטלפון.
   המחשב — נוחות, לא הכרח.
3. **אין תשתית לפני הוכחה.** Supabase (מחובר) ו-Vercel (זמין) נכנסים רק בשלב 5, אחרי שה-POC עובר.
   הסיבה נשארת כמו ב-v4: היישור דורש GPU ודקות ריצה — Vercel serverless לא מסוגל להריץ אותו ממילא;
   מה ש-Vercel/Supabase כן נותנים הוא ממשק העלאה ואחסון, וזה רלוונטי רק כשיש pipeline שעובד.

## 3. ה-Pipeline (מעודכן)

```
1. טקסט     → Sefaria API v3, גרסת MAM (עם טעמים)
2. ניקוי    → הסרת HTML/entities/{פ}{ס}                          ← חדש ב-v5
3. חיתוך    → פיצול רקורסיבי לפי דרגות הטעמים עד ≤4 מילים
              (dedupe טעמי-עזר, שרשראות מקף, שובר-שוויון, פולבק)  ← חדש ב-v5
4. הכנה     → שני וריאנטים של alignment_text:                     ← שונה מ-v4
              א. מנוקד (ל-MMS+uroman)   ב. אותיות בלבד (ל-wav2vec2 עברי)
              בשניהם: קרי במקום כתיב + יהוה→אֲדֹנָי (קרי-קבוע)
5. יישור    → CTC על (אודיו 16kHz מונו + alignment_text) → זמן לכל מילה
6. דגימה    → רק גבולות מקטעים; שאר החותמות נזרקות
7. ייצוא    → SRT: display_text + הזמנים משלב 6
8. עריכה    → CapCut Web (ייבוא, עיצוב, תיקון ידני של 2–3 מקטעים) → Cloud → מובייל → פרסום
```

**האינווריאנט הקריטי (ירושה מ-v4, עדיין נקודת הכשל הצפויה בקוד):**
display_text ו-alignment_text חייבים להיות מיושרים אינדקס-לאינדקס ברמת המילה.

## 4. אלגוריתם החיתוך (מפרט מחייב — ממומש ב-`src/chunker`)

```
פצל(יחידה):
    אם מספר קבוצות-המקף ביחידה <= 4:  החזר כמקטע אחד
    אחרת:
        מצא את הטעם המפסיק בדרגה החזקה ביותר ביחידה
        אם יש כמה באותה דרגה: קח את הקרוב ביותר לאמצע היחידה
        אם אין אף מפסיק: חלק שווה-בשווה על גבולות קבוצות-מקף
        חתוך אחרי המילה נושאת-הטעם; פצל(חלק ראשון); פצל(חלק שני)
```

| דרגה | טעמים | קודפוינטים |
|---|---|---|
| 1 | סוף-פסוק, אתנחתא | U+05C3, U+0591 |
| 2 | סגולתא, שלשלת, זקף קטן, זקף גדול, טפחא | U+0592, U+0593, U+0594, U+0595, U+0596 |
| 3 | רביע, זרקא (שני וריאנטים!), פשטא, יתיב, תביר | U+0597, **U+0598 / U+05AE**, U+0599, U+059A, U+059B |
| 4 | גרש, גרשיים, קרני פרה, תלישא גדולה, פזר | U+059C, U+059E, U+059F, U+05A0, U+05A1 |
| — | משרתים — לעולם לא נקודת חיתוך | מונח, מהפך, מרכא, מרכא כפולה, דרגא, קדמא, תלישא קטנה, ירח בן יומו |

כללים קשיחים: לא חותכים על U+05BD (מתג≠סילוק); לא חותכים בתוך שרשרת מקף; ספירת מילים על
אותיות בלבד אחרי ניקוי; טעם שמופיע פעמיים על מילה (עזר של MAM) נספר פעם אחת.

**מבחן קבלה (כמו ב-v4):** הרצה על פרק שלם, קריאת המקטעים בעיניים — חיתוכים בגבולות תחביריים הגיוניים.

## 5. שלבי העבודה

### שלב 0 — אימותים (notebook מוכן, ~20 דק' מהטלפון)
לפתוח את `notebooks/shelav0_verification.ipynb` ב-Colab, למלא תא-פרמטרים (ספר, פרק, נתיב ההקלטה
ב-Drive), ‏Run all. ה-notebook מדפיס בעברית: אימות גרסת MAM וטעמיה, טבלת תדירויות (מכריעה את שאלת
הזרקא), קידוד קרי/כתיב בפועל, מקטעי החיתוך לבדיקת עיניים, ופרטי ההקלטה (פורמט/קצב/ערוצים/משך).
**תנאי מעבר:** יש טעמים בטקסט; החיתוך נראה הגיוני; ההקלטה נקראת.

### שלב 1 — פונקציית החיתוך ✅ (נבנתה, בריפו, עם בדיקות)
נשאר רק האימות מול טקסט MAM אמיתי (חלק משלב 0), כי סביבת הפיתוח חסומה לרשת.

### שלב 2 — ה-POC (יום–יומיים)
השאלה היחידה (ירושה מ-v4): האם יישור CTC נותן גבול-מקטע בדיוק ~1 שנייה על קריאה בטעמים אמיתית?
1. ההקלטה הקיימת — בטעמים, מהמיקרופון האמיתי. לא קריאה ישרה.
2. ground truth: סימון ידני של גבולות מקטעים. מהטלפון אין Audacity — חלופה: תא ב-notebook שמנגן
   כל מקטע-מועמד ומאפשר תיקון, או סימון במחשב אם זמין (נוחות, לא הכרח).
3. הרצת **שלושה** מתחרים: MMS_FA+uroman(מנוקד) / wav2vec2-עברי(לא מנוקד) / baseline חלוקה-שווה.
4. מדד: התפלגות שגיאת נקודת-כניסה, בחתך מוקדם/מאוחר.
**תנאי מעבר:** ~90% מהמקטעים בטווח ~1 שנייה, ובפער משמעותי מעל ה-baseline.

### שלב 3 — ייצוא SRT + בדיקת CapCut (חצי יום, רק אם שלב 2 עבר)
UTF-8, ‏`HH:MM:SS,mmm --> HH:MM:SS,mmm`, טקסט = display_text.
**בדיקת האמת מהטלפון:** ייבוא ב-CapCut Web (דפדפן טלפון במצב "אתר מחשב" — לאמת שזה עובד; אם לא,
זו הרצה חד-פעמית מהמחשב) → לוודא שהניקוד והטעמים שרדו → Cloud Sync → פתיחה באפליקציית המובייל.

### שלב 4 — זרימת עבודה מלאה מהטלפון
הקלטה בטלפון → העלאה ל-Drive → Colab (Run all) → הורדת SRT → CapCut Web → Cloud → מובייל → פרסום.

### שלב 5 — שירות טלפוני מלא (רק אחרי ששלב 4 עבד פעמיים-שלוש בידיים)
מטרה: להעלים את Colab מהזרימה היומיומית.
- **Vercel:** דף אחד בעברית — בחירת ספר/פרק, העלאת הקלטה, כפתור "עבד", קישור להורדת SRT.
- **Supabase (כבר מחובר):** אחסון הקלטות ו-SRT, טבלת עבודות (סטטוס: ממתין/רץ/מוכן/שגיאה).
- **עיבוד:** worker עם GPU לפי-שימוש (Modal / RunPod / HF — ההשוואה תיעשה אז, בעלות של סנטים לריצה).
  Vercel לא מריץ את היישור — הוא רק הדלפק.
**לא מקימים כלום מזה עכשיו.** ההחלטה תקבל מסמך קצר משלה כששלב 4 יוכיח את עצמו.

## 6. סיכונים — במפורש

1. **הסיכון המרכזי:** מודלי ASR אומנו על דיבור; קריאה בטעמים היא חצי-שירה (הברות מוארכות, מליסמות,
   הפסקות ארוכות בסוף פסוק). ייתכן שהיישור יסטה דווקא במקומות המוזיקליים. לכן ה-POC קודם לכל,
   ולכן ה-baseline חובה. גם אם ה-POC נכשל — **עדיין יש מוצר** (טקסט מדויק + חלוקה שווה + גרירה ידנית).
2. **יישור נכשל בביטחון גבוה** (אזהרת v1, בתוקף): המיישר תמיד מחזיר תשובה. בקרת איכות נדחית לאחרי
   ה-POC, אבל "המשתמש רואה ומתקן" היא תשובה לגיטימית למשתמש יחיד.
3. **CapCut:** אין ייבוא SRT במובייל (נכון ל-2026) — המעקף דרך Web+Cloud. אם גם CapCut Web מדפדפן
   טלפון לא יעבוד, זו הנקודה היחידה בזרימה שדורשת מחשב — עד שלב 5, ששם אפשר יהיה לשקול חלופות.
4. **קרי/כתיב:** אם הקידוד ב-MAM יתברר כמסובך לחילוץ אוטומטי, פתרון ביניים: רשימת החלפות ידנית
   לפרקים שמקליטים בפועל (מעטים בכל שבוע).
